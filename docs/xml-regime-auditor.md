# XML Regime Auditor

`scripts/audit-xml-regime.ps1` compares an expected CNPJ tax-regime list with recent emitted NF-e/NFC-e XML evidence, with basic national NFS-e and CT-e support.

This is an operational audit helper. It identifies likely mismatches between expected tax-regime classification and emitted XML evidence. It is not legal or accounting advice.

## Supported Evidence

Version 1 supports outbound:

- `NF-e`
- `NFC-e`
- `NFS-e` when the XML exposes national layout fields such as `opSimpNac`
- `CT-e` as medium-confidence evidence from ICMS structure

It intentionally ignores `Entradas` by default because incoming XMLs describe suppliers, not the company whose folder is being audited.

MDF-e is not used as direct regime evidence because the observed layouts reference other documents but do not expose regime fields. NFS-e layouts vary by municipality, so unsupported municipal layouts may still produce `UnknownXmlRegime`.

## Expected CSV

Minimal public format:

```csv
cnpj;expected_regime
00000000000191;Simples Nacional
00000000000272;Normal
```

Supported CNPJ column names:

- `cnpj`
- `CNPJ`
- `cnpj_normalizado`
- `documento`

Supported regime column names:

- `expected_regime`
- `regime_tributario`
- `regime`
- `tipo_regime`

Expected regime normalization:

- `MEI`, `Simples`, `Simples Nacional` -> Simples family
- `Normal`, `Regime Normal`, `Lucro Presumido`, `Lucro Real` -> Normal family

The expected CSV is treated as the expected current regime at audit time. The tool does not validate historical regime effective dates. A mismatch is a review signal, not proof of fiscal error.

## XML Folder Shape

The auditor expects:

```text
<XmlRoot>\<CNPJ>\<tipo>\<yyyyMM>\<optional nested folders>\*.xml
```

Example synthetic fixture:

```text
examples\xml-regime\00000000000191\NF-e\202604\Caixa1\simples-ok-procNFe.xml
```

## Classification Rules

Primary evidence is `<emit><CRT>`:

| CRT | XML regime |
|---|---|
| `1` | Simples Nacional |
| `2` | Simples Nacional - Excesso de Sublimite |
| `3` | Regime Normal |

Secondary ICMS evidence is used only when `CRT` is missing, or to flag internal XML conflicts:

- `ICMSSN*` or `CSOSN` -> Simples evidence
- `ICMS00`, `ICMS10`, etc. or ICMS-level `CST` -> Normal evidence

PIS/COFINS `CST` is not treated as ICMS regime evidence.

For NFS-e, `opSimpNac` is treated as medium-confidence evidence when no NF-e-style `CRT` exists:

- `2` or `3` -> Simples evidence
- `1` -> Normal evidence

For CT-e, the tool treats `ICMSSN` or `indSN=1` as Simples evidence and normal ICMS groups such as `ICMS00` plus ICMS-level `CST` as Normal evidence. CT-e does not carry NF-e-style `CRT`, so this is lower confidence than NF-e/NFC-e evidence.

The emitter CNPJ in `<emit><CNPJ>` must match the CNPJ folder. If it does not, the row is marked for review with `IssuerMismatch`.

## Status Contract

The report uses a small status set:

- `OK`
- `Mismatch`
- `Review`
- `NoEvidence`
- `Error`

The `reason` column contains the operational detail, such as:

- `MatchedExpectedRegime`
- `ExpectedSimplesXmlNormal`
- `ExpectedNormalXmlSimples`
- `Sublimite`
- `XmlInternalConflict`
- `IssuerMismatch`
- `NoRecentXml`
- `UnknownExpectedRegime`
- `ParseError`

## Usage

Public sample run:

```powershell
.\scripts\audit-xml-regime.ps1 `
  -XmlRoot '.\examples\xml-regime' `
  -ExpectedCsvPath '.\examples\expected-regimes.sample.csv' `
  -OutputPath '.\output\xml-regime-audit.csv'
```

Show only actionable rows:

```powershell
.\scripts\audit-xml-regime.ps1 `
  -XmlRoot '.\examples\xml-regime' `
  -ExpectedCsvPath '.\examples\expected-regimes.sample.csv' `
  -OutputPath '.\output\xml-regime-problems.csv' `
  -OnlyProblems
```

Generate Portuguese problem reports grouped by divergence type and XML type:

```powershell
.\scripts\audit-xml-regime.ps1 `
  -XmlRoot '.\examples\xml-regime' `
  -ExpectedCsvPath '.\examples\expected-regimes.sample.csv' `
  -OutputPath '.\output\xml-regime-audit.csv' `
  -ProblemReportPath '.\output\xml-regime-problemas.csv' `
  -GroupedReportPath '.\output\xml-regime-grupos.csv'
```

The grouped report uses labels such as:

- `Simples com XML Normal`
- `Normal com XML Simples`
- `XML Simples com Excesso de Sublimite`
- `CNPJ do emitente diferente da pasta`
- `Sem XML recente nos tipos auditados`

CSV files generated by `scripts/classify-clientes.ps1` can be used directly as `-ExpectedCsvPath` when they contain `cnpj` or `cnpj_normalizado` and `regime_tributario`. The auditor also auto-detects comma-separated CSVs generated by spreadsheet/report tools.

Internal/network-share usage should pass the real XML root explicitly:

```powershell
.\scripts\audit-xml-regime.ps1 `
  -XmlRoot '\\your-server\XML' `
  -ExpectedCsvPath 'C:\path\clientes_classificados.csv' `
  -OutputPath 'C:\path\xml-regime-audit.csv' `
  -IncludeTypes 'NF-e,NFC-e,NFS-e' `
  -ThrottleLimit 8 `
  -OnlyProblems
```

## Performance Controls

- `-LatestMonthsToTry 3`: tries only the newest month folders per invoice type.
- `-MaxFilesPerCompanyType 3`: parses only a limited number of recent XML candidates per company/type/month.
- `-IncludeTypes @('NF-e','NFC-e')`: limits outbound document types.
- `-ThrottleLimit 8`: processes companies in bounded parallel chunks on Windows PowerShell.

The script avoids global recursive scans. It loads expected CNPJs first, then scans only matching company folders.

## Privacy

Do not commit real XMLs, expected-regime CSVs, generated reports, cache files, client spreadsheets, or company data. Use only synthetic fixtures under `examples/`.
